# Restore Drill

This document describes how to prove backups are usable for the single-VM Docker
Compose deployment.

Run restore drills on a disposable VM, disposable disk, or temporary local
Compose stack whenever possible. Restoring directly over production data should
be a last resort.

`AIS_DEPLOY_USE_SUDO_DOCKER` only applies to project scripts that support it,
such as `scripts/deploy/smoke-check.sh`. For raw Docker commands shown in this
document, prefix the command with `sudo` manually if your VM user is not in the
`docker` group.

## Backup Files

Postgres backups are custom-format `pg_dump` files:

```text
/opt/ais-tracking-system/backups/postgres/postgres-YYYYMMDDTHHMMSSZ.dump
```

Redis backups are compressed archives of the Redis `/data` directory:

```text
/opt/ais-tracking-system/backups/redis/redis-YYYYMMDDTHHMMSSZ.tar.gz
```

Treat both as private data.

## Compose Helper

Use this shell helper for the restore commands below. It includes the optional
Grafana-local override only when the file exists.

```bash
cd /opt/ais-tracking-system

COMPOSE_FILES=(-f docker-compose.prod.yml)
if [[ -f docker-compose.prod.grafana-local.yml ]]; then
  COMPOSE_FILES+=(-f docker-compose.prod.grafana-local.yml)
fi

compose() {
  docker compose \
    --env-file .env.production \
    --env-file .env.release \
    "${COMPOSE_FILES[@]}" \
    "$@"
}
```

If your VM user is not in the `docker` group, replace `docker compose` in the
helper with `sudo docker compose`.

## Postgres Restore Drill

1. Copy the backup to the drill machine.
2. Start the production Compose stack with a disposable database volume.
3. Stop app services so they do not write during restore.

```bash
compose stop api ingestion worker nginx prometheus grafana
```

4. Copy the backup into the Postgres container.

```bash
POSTGRES_BACKUP=/opt/ais-tracking-system/backups/postgres/postgres-latest.dump
POSTGRES_CONTAINER="$(compose ps -q postgres)"
docker cp "$POSTGRES_BACKUP" "$POSTGRES_CONTAINER:/tmp/postgres-restore.dump"
```

Use `sudo docker cp` if your VM user needs sudo for Docker.

5. Drop and recreate the app database inside the disposable environment.

```bash
compose exec -T postgres sh -c 'dropdb -U "$POSTGRES_USER" "$POSTGRES_DB" --if-exists'
compose exec -T postgres sh -c 'createdb -U "$POSTGRES_USER" "$POSTGRES_DB"'
```

6. Restore the backup.

```bash
compose exec -T postgres sh -c 'pg_restore -U "$POSTGRES_USER" -d "$POSTGRES_DB" --no-owner --no-privileges /tmp/postgres-restore.dump'
```

The database was dropped and recreated immediately before restore, so
`--clean --if-exists` is intentionally omitted here. If you deliberately restore
into a non-empty database during a separate drill, add cleanup flags only after
you understand what will be dropped.

7. Run DB-level sanity checks.

These checks verify the restored schema is readable. They do not require rows,
because a valid backup may come from a low-data or empty environment.

```bash
compose exec -T postgres sh -c 'psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" -c "\dt"'
compose exec -T postgres sh -c 'psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" -c "select count(*) from vessels;"'
compose exec -T postgres sh -c 'psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" -c "select count(*) from vessel_positions_latest;"'
```

8. Start services and run smoke checks.

```bash
compose up -d
AIS_DEPLOY_USE_SUDO_DOCKER=true scripts/deploy/smoke-check.sh
```

Expected result: services start, API responds, DB sanity checks pass, and no
missing-table errors appear in logs.

## Redis Restore Drill

Redis uses AOF persistence in production. The Redis backup script archives the
entire `/data` directory so it can preserve AOF and RDB files generated by
Redis.

Redis is operationally important, but it is not the same durability layer as
Postgres. It may contain streams, pending consumer state, BullMQ state,
deduplication or sampling keys, and enrichment coordination keys. After Redis
loss or restore, Postgres remains the source of truth. The live AIS feed can
repopulate current activity over time, but derived async work may need
reconciliation or backfill, and exact in-flight stream state may not be
perfectly recoverable.

1. Stop services that use Redis, including Redis itself.

```bash
compose stop api ingestion worker redis
compose ps redis
```

Do not clear or restore the Redis volume while Redis is running.

2. Select the Redis volume from the current Compose project.

```bash
REDIS_BACKUP=/opt/ais-tracking-system/backups/redis/redis-latest.tar.gz
REDIS_CONTAINER="$(compose ps -aq redis)"

if [[ -z "$REDIS_CONTAINER" ]]; then
  echo "Could not resolve the Compose redis container" >&2
  exit 1
fi

REDIS_VOLUME="$(docker inspect --format '{{range .Mounts}}{{if eq .Destination "/data"}}{{.Name}}{{end}}{{end}}' "$REDIS_CONTAINER")"

if [[ -z "$REDIS_VOLUME" ]]; then
  echo "Could not resolve Redis /data volume from the Compose redis service" >&2
  exit 1
fi

echo "Selected Redis volume: $REDIS_VOLUME"
read -r -p "Type YES to erase and restore this Redis volume: " confirm
if [[ "$confirm" != "YES" ]]; then
  echo "Redis restore cancelled"
  exit 1
fi
```

If your VM user needs sudo for Docker, use:

```bash
REDIS_VOLUME="$(sudo docker inspect --format '{{range .Mounts}}{{if eq .Destination "/data"}}{{.Name}}{{end}}{{end}}' "$REDIS_CONTAINER")"
```

3. Restore the archive into the selected Redis volume in a disposable
environment.

```bash
REDIS_BACKUP_BASENAME="$(basename "$REDIS_BACKUP")"

docker run --rm \
  -v "$REDIS_VOLUME:/data" \
  -v "$(dirname "$REDIS_BACKUP"):/backup:ro" \
  -e REDIS_BACKUP_BASENAME="$REDIS_BACKUP_BASENAME" \
  alpine:3.20 \
  sh -c 'rm -rf /data/* && tar -C /data -xzf "/backup/$REDIS_BACKUP_BASENAME"'
```

Use `sudo docker run` if your VM user needs sudo for Docker.

4. Restart Redis and app services.

```bash
compose up -d redis
compose up -d api ingestion worker nginx prometheus grafana
AIS_DEPLOY_USE_SUDO_DOCKER=true scripts/deploy/smoke-check.sh
```

Expected result: Redis starts cleanly and application services reconnect.

## Production Restore Caution

Before restoring production:

- take fresh Postgres and Redis backups first;
- stop app writers;
- preserve the current Docker volumes until the restore is confirmed;
- expect database rollback after schema-changing migrations to require manual
  judgment;
- record the backup filenames and commands used.
